남의 코드로 배워보는 Design Pattern (1)

Posted on May 11, 2018

Design patterns implemented in Java 를 둘러보면서 현업에서 겪었던 또는 그 비슷한 이야기를 풀어보자.

Adapter

설명은 링크에 잘 나와있으니까 그쪽을 참고하는게 좋다.

also known as __Wrapper__ 말 그대로 A에서 B를 쓰는데 C로 감싸서 A->C(B) 요렇게 사용한다는 것.

개인적인 경험으로 많이 활용되었던 2가지 경우는

  1. 3rd party 나 black box를 변경없이 사용하려고 할 때
  2. 같은 기능으로 여러 3rd party를 지원해야 할 경우

정도였다.

Balking

한마디로 IllegalStateException.
상태 값을 관리해서 괜찮을 때가 아니면 Balking(=실패) 하는 것.
상태 값을 실시간으로 관리하는 상황이 많으므로 필연적으로 구현에 동기화 문제를 고려해야한다.
애초에 상태 관리는 고려할 게 많아서 정말 간단하게 만들던지, 유한 상태 기계 (Finite State Machine) 같은 걸 이용해서 확실히 관리 하던지 해야한다.

Callback

이 것도 패턴이라면 패턴인건가? 너무나도 기본적인 개발 기법.
처리 로직을 분리시킬 경우, 분리된 로직이 완료될 때 등록된 로직을 수행하게 한다. 요때 등록된 로직을 callback이라고 한다. 일일이 열거하기 힘들정도로 쓰는데가 너무 많다.
IO나 GUI 등 비동기 처리를 할 때 주로 쓴다.

Converter

데이터 형으로 사용되는 객체를 비슷하지만 조금 다른 형태로 변환하고자 할 때, 변환 로직을 아예 별도 객체로 만들어두는 것.
네트워크 전송을 위한 객체(DTO)를 사용할 때 주로 쓴다.

Data Access Object

db나 disk 같은 영속 저장소에 접근할 때, 접근 계층을 추상화해서 접근 로직을 유연하게 하는 것.
현재 존재하고 향후 추가될 다양한 저장소 접근 로직 구현과 비즈니스 로직을 분리하는 것으로 확장성을 높인다… 가 주요 목적임.

access layer를 interface로 빼고 implements 와 독립성을 갖게 하는건 객체 지향 개발의 장점이자 정말 일반적인 활용법.
